# 需求评审检查清单 — requirements-review

## 完整性检查

- [ ] 每个功能有对应的用户故事
- [ ] 每个用户故事有验收标准
- [ ] 验收标准可测试（有明确的通过/不通过条件）
- [ ] 边界条件已定义（最大值、最小值、异常输入）
- [ ] 错误处理已描述
- [ ] 非功能需求已量化
- [ ] 依赖已列出
- [ ] 优先级已分配

## 质量评估

### 好的需求
- **具体** — "电池续航 ≥ 30 分钟（25°C 无风环境）"
- **可测试** — QA 能据此写出测试用例
- **唯一解释** — 只有一种理解
- **独立** — 不依赖其他需求的实现

### 坏的需求
- **模糊** — "系统应该快速响应"（多快？）
- **不可测试** — "用户体验应该良好"（怎么评判？）
- **歧义** — "支持高速飞行"（多快？什么算高速？）
- **耦合** — "在 A 功能完成后才能定义 B"

## 无人机特定关注点

1. **飞行安全** — 所有安全相关需求必须 P0
2. **法规** — 频率、高度、重量是否符合当地法规
3. **环境** — 操作环境范围是否明确
4. **电池** — 低电量行为是否明确定义
5. **通信** — 失联/断连行为是否定义
6. **载荷** — 云台/相机的规格是否与飞行性能匹配
