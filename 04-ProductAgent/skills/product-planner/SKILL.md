# 产品规划 — product-planner

## 产品初始化

当用户表示要新建一个产品时，执行以下步骤：

1. 确认产品名称（如「无人机运营服务平台」）
2. 创建 workspace 目录结构：
```
workspace/{产品名称}/
  .product-info.json
  00-reference/    ← 参考文档（行业资料、竞品资料、历史文档）
  01-market/
  02-strategy/
  03-requirements/
  04-prototype/
  05-roadmap/
```
3. 初始化 `.product-info.json`：
```json
{
  "name": "{产品名称}",
  "created": "YYYY-MM-DD",
  "current_phase": 1,
  "completed_phases": [],
  "phases": {
    "1": { "name": "看清市场", "status": "in_progress", "artifacts": [] },
    "2": { "name": "想清策略", "status": "not_started", "artifacts": [] },
    "3": { "name": "定好需求", "status": "not_started", "artifacts": [] },
    "4": { "name": "画好原型", "status": "not_started", "artifacts": [] },
    "5": { "name": "排期指标", "status": "not_started", "artifacts": [] }
  }
}
```
4. 告知用户产品已创建，展示当前进度，引导进入 P1 阶段。

## 五阶段流程

### P1 — 看清市场（对应命令 `/market`）

**前置条件：** 无

**检查清单：**
- [ ] 明确产品要解决什么问题
- [ ] 识别目标客户类型（B端 / G端 / 混合）
- [ ] 至少分析 3 个直接竞品或替代方案
- [ ] 明确目标客户的核心痛点（不少于 3 个）

**产出文档：**
| 文档 | 模板 | 输出位置 |
|---|---|---|
| 竞品分析报告 | 见下方模板 | `01-market/竞品分析.md` |
| 用户画像 | 见下方模板 | `01-market/用户画像.md` |
| 市场规模测算 | 见下方模板 | `01-market/市场规模.md` |
| MRD 产品立项文档 | 见下方模板 | `01-market/MRD.md` |

**前置文档（为 P2 提供输入）：** 竞品分析报告 + 用户画像 + MRD

---

### P2 — 想清策略（对应命令 `/market business`）

**前置条件：** P1 产出物

**检查清单：**
- [ ] 有明确的产品定位声明
- [ ] 商业模式画布已填写
- [ ] 差异化优势可被用户感知

**产出文档：**
| 文档 | 模板 | 输出位置 |
|---|---|---|
| 产品定位 | 见下方模板 | `02-strategy/产品定位.md` |
| 商业模式 | 见下方模板 | `02-strategy/商业模式.md` |

**前置文档（为 P3 提供输入）：** 产品定位 + 商业模式

---

### P3 — 定好需求（对应命令 `/requirements`）

**前置条件：** P2 产出物（至少需要产品定位）

**检查清单：**
- [ ] 用户角色和权限体系已定义
- [ ] 功能清单覆盖核心业务流程
- [ ] 每个需求有明确的优先级
- [ ] 用户故事不少于 5 条

**产出文档：**
| 文档 | 模板 | 输出位置 |
|---|---|---|
| BRD 业务需求文档 | 见下方模板 | `03-requirements/BRD.md` |
| PRD 产品需求文档 | 见下方模板 | `03-requirements/PRD.md` |
| 用户故事/用例 | 见下方模板 | `03-requirements/用户故事.md` |
| 需求池 | 见下方模板 | `03-requirements/需求池.md` |

**前置文档（为 P4 提供输入）：** PRD + 用户故事

---

### P4 — 画好原型（对应命令 `/prototype`）

**前置条件：** P3 产出物

**检查清单：**
- [ ] 信息架构覆盖 PRD 中的核心功能
- [ ] 每个用户角色有对应的操作流程
- [ ] 关键交互流程有说明（不少于 3 个）

**产出文档：**
| 文档 | 模板 | 输出位置 |
|---|---|---|
| 信息架构 | 见下方模板 | `04-prototype/信息架构.md` |
| 交互原型说明 | 见下方模板 | `04-prototype/交互原型.md` |
| 页面清单 | 见下方模板 | `04-prototype/页面清单.md` |

**前置文档（为 P5 提供输入）：** 信息架构 + 交互原型

---

### P5 — 排期指标（对应命令 `/roadmap` + `/metrics`）

**前置条件：** P4 产出物

**检查清单：**
- [ ] 版本规划有明确的里程碑时间点
- [ ] MVP 范围清晰（不超过核心功能的 40%）
- [ ] 北极星指标已定义
- [ ] 每个阶段有可衡量的成功标准

**产出文档：**
| 文档 | 模板 | 输出位置 |
|---|---|---|

## 进度管理

### 产品状态文件更新规范

`.product-info.json` 是产品的唯一状态源。**每次产出后立即更新**，不要等用户提醒。

需要更新的触发场景：
- 生成新文件（PRD、竞品分析、原型等）
- 用户提供新的背景信息或补充需求
- 讨论中形成关键决策或结论
- 完成一个阶段或阶段内子任务

更新内容：
```json
{
  "name": "{产品名称}",
  "description": "{一句话描述}",
  "created": "YYYY-MM-DD",
  "current_phase": N,
  "completed_phases": [1, 2],
  "phases": {
    "1": {
      "name": "看清市场",
      "status": "completed | in_progress | not_started",
      "artifacts": ["01-market/竞品分析.md", "01-market/用户画像.md"],
      "notes": "关键决策或备注（可选）"
    }
  },
  "decisions": [
    { "date": "2026-04-21", "decision": "PRD 覆盖全年 5 大模块而非仅 V2.0", "rationale": "补齐 V1.0 缺失的过程文档" }
  ]
}
```

更新方式：
1. 读取当前 `.product-info.json`
2. 追加新 artifacts 到对应阶段
3. 如阶段全部 artifacts 完成，标记 status 为 completed，current_phase +1
4. 写入文件

### 阶段推进规则

每次完成一个阶段后：

1. 告知用户当前进度
2. 提示下一阶段需要的前置文档（如已准备好则引导进入）

如果用户在执行某个命令时发现前置阶段未完成，应：
1. 提醒用户前置阶段的风险
2. 如果用户确认要跳过，仍然执行但做记录
3. 建议在后续补上前置阶段
