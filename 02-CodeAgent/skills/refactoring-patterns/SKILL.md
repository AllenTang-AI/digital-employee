# 重构模式 — refactoring-patterns

## 微重构（安全，PR 级别）

### 提取函数
**何时用：** 代码块有清晰的单一职责，可以命名。
**怎么做：** 复制代码块 → 创建有描述性名称的新函数 → 替换为调用。
**风险：** 低 — 行为不变。

### 条件替换为多态
**何时用：** switch/if 链检查类型并分发不同行为。
**怎么做：** 创建接口 → 按类型实现 → 用接口调用替换 switch。
**风险：** 中 — 需要充分测试。

### 提取接口
**何时用：** 一个类有明确的方法子集被特定客户端使用。
**怎么做：** 定义接口 → 类实现它 → 客户端依赖接口。
**风险：** 低 — 行为不变。

### 分解条件表达式
**何时用：** 复杂的布尔条件难以理解。
**怎么做：** 提取子表达式为命名函数/变量。
**风险：** 低 — 纯提取操作。

## 宏重构（需要规划）

### 拆分模块
**何时用：** 模块超过 500 行或有多个职责。
**怎么做：** 识别内聚组 → 创建新模块 → 移动代码 → 更新导入。
**风险：** 高 — 需要全面的测试覆盖。

### 引入分层
**何时用：** 业务逻辑与基础设施（数据库、HTTP、文件 I/O）混在一起。
**怎么做：** 定义层边界 → 提取领域逻辑 → 为基础设施创建适配器。
**风险：** 高 — 影响很多文件。

### 全局变量替换为依赖注入
**何时用：** 全局状态导致测试困难或隐藏耦合。
**怎么做：** 识别全局使用点 → 改为构造函数/参数传递 → 移除全局。
**风险：** 中 — 可能需要更新多处调用点。

## 安全规则

1. **一个 PR 只做一种重构** — 不要混重构和功能变更
2. **测试先行** — 重构前确保有测试；如果没有，先加测试
3. **小提交** — 每个提交都应该能独立编译
4. **分两步重命名** — 旧名称 → 标记废弃 → 下次发布时移除
